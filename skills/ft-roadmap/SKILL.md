# /ft:roadmap — 로드맵 기획

프로젝트/패키지 단위의 큰 그림을 잡고,
Pillar/모듈별 Feature.md를 일괄 생성한다.

## 왜 이 스킬이 필요한가

/ft:plan은 피처 1개의 Feature.md만 만든다.
하지만 새 패키지나 대규모 기능을 시작할 때는
"어떤 피처들이 필요하고, 어떤 순서로 만들지"를 먼저 잡아야 한다.
이 상위 계획 없이 개별 피처를 바로 /ft:build 할 수 없다.

/ft:roadmap은 이 간극을 메운다:
- TASK.md, BRAINSTORM.md 같은 기획 문서를 읽고
- 피처 단위로 쪼개서 Feature.md × N개를 일괄 생성하고
- 의존성 기반 구현 순서를 제안한다.

## /ft:plan과의 차이

| | ft:plan | ft:roadmap |
|---|---------|------------|
| 단위 | 피처 1개 | 프로젝트/패키지 전체 |
| 출력 | Feature.md 1개 | Feature.md N개 + 구현 순서 |
| 기획 문서 | 참조 안 함 | TASK.md/BRAINSTORM.md 자동 수집 |
| 사용자 확인 | Feature.md마다 확인 | 분류 제안에서 1회 확인 → 일괄 생성 |
| 의존성 | 기존 피처 간 | Pillar 간 의존성 + 구현 순서 도출 |

## 실행 흐름

### 0단계: 모호성 체크

사용자의 요청에서 아래 3가지가 파악되는지 확인한다:

1. **목표** — 뭘 만드는 프로젝트/패키지인지 (예: "Addressables 확장 패키지")
2. **범위** — 어디까지 만드는지 (예: "런타임 안전 + 디버깅 + 검증")
3. **성공 기준** — MVP 완성의 정의 (예: "SafeHandle로 로드/해제 + 누수 감지 동작")

3가지 중 하나라도 빠지면:
- 빠진 항목에 대해 질문한다
- 질문은 한 번에 하나씩, 목표부터
- 3가지가 모두 파악되면 1단계로 진행

### 1단계: 기획 문서 수집

**프로젝트 루트부터** 기획 관련 문서를 탐색한다.
대상 패키지 폴더만 보면 안 된다 — 기획 문서는 별도 폴더에 있는 경우가 많다.
(예: `Assets/Address/TASK.md`가 `Assets/AddrX/` 패키지의 기획 문서일 수 있다)

#### 탐색 순서

1. **프로젝트 루트에서 Glob 검색** (대상 패키지에 한정하지 않는다):
   ```
   **/TASK.md
   **/BRAINSTORM.md
   **/RESEARCH.md
   **/README.md
   **/Feature.md
   .featurecraft/FEATURE_INDEX.md
   ```

2. **관련 폴더 탐색** — 대상 패키지와 이름이 유사한 폴더가 있으면 반드시 확인:
   - 예: 대상이 `AddrX`이면 → `Address`, `Addressable`, `Addr` 등도 탐색
   - 예: 대상이 `SupaRun`이면 → `GameServer`, `Server` 등도 탐색
   - 이런 폴더에 TASK.md, BRAINSTORM.md 등이 있으면 기획 문서로 간주

3. **발견된 문서를 읽고 핵심 정보를 추출한다**:
   - 대상 패키지와 관련 있는 내용만 필터링
   - 각 문서에서 피처 목록, 의존성, 우선순위 정보 추출

- 문서가 하나도 없으면 → 사용자에게 직접 요구사항을 질문한다
- 문서가 있으면 → "이 문서들을 기반으로 로드맵을 잡겠습니다" 안내

### 2단계: 피처 분류 제안

기획 문서 + 사용자 요구사항을 기반으로 피처 묶음을 제안한다.

#### 적정 피처 크기 판단 기준

| 기준 | 너무 작음 | 적정 | 너무 큼 |
|------|-----------|------|---------|
| 파일 수 (예상) | 1~2개 | 3~10개 | 20개+ |
| 독립 이해 가능 | 다른 것과 합쳐야 이해됨 | 단독으로 이해/구현 가능 | 여러 책임 혼재 |
| 구현 단위 | 함수 하나 | 모듈/폴더 하나 | 프로젝트 절반 |

#### 제안 포맷

```
## 로드맵 제안: {프로젝트명}

### Pillar 1: {이름} — {한줄 설명}
포함 기능: A, B, C
의존성: 없음 (기반)
예상 위치: {경로}

### Pillar 2: {이름} — {한줄 설명}
포함 기능: D, E, F
의존성: Pillar 1
예상 위치: {경로}

...

### 구현 순서
1. Pillar 1 (기반, 의존성 없음)
2. Pillar 2 (Pillar 1 의존)
3. Pillar 3 (독립)
4. Pillar 4 (Pillar 1, 3 의존)
```

### 3단계: 사용자 확인

- 제안된 피처 분류를 사용자에게 보여준다
- 피드백을 받아 조정한다:
  - 피처 합치기 / 쪼개기
  - 이름 변경
  - 포함 기능 이동
  - 구현 순서 변경
  - 범위 축소/확장
- **사용자가 "좋아" / 확인하면 4단계로 진행**

### 4단계: Feature.md 일괄 생성

확정된 피처 분류에 따라 Feature.md를 한번에 생성한다.
(ft:plan처럼 하나씩 확인받지 않는다 — 3단계에서 이미 확인받았으므로)

각 Feature.md 포맷:

```markdown
# {피처명}

## 상태
wip

## 용도
{한 줄 설명}

## 의존성
- {상대경로} — {참조 이유}

## 포함 기능
- {기능 1} — {설명}
- {기능 2} — {설명}
- ...

## 구조
(아직 없음 — /ft:build 후 업데이트)

## API (외부 피처가 참조 가능)
(아직 없음 — /ft:build 후 업데이트)

## 주의사항
- {있으면 기재}
```

**ft:plan의 Feature.md와 차이점:**
- `## 포함 기능` 섹션 추가 — 이 피처 안에 들어갈 세부 기능 목록
- /ft:build가 이 목록을 보고 구현 범위를 파악

### 5단계: 마무리

1. **FEATURE_INDEX.md 갱신** — 생성된 모든 피처 등록
2. **구현 순서 안내** — "다음 단계: `/ft:build {첫 번째 피처 경로}`로 구현을 시작하세요."
3. **생성 결과 요약** 출력:

```
## 로드맵 생성 완료

| # | 피처 | 경로 | 의존성 |
|---|------|------|--------|
| 1 | Core | Runtime/Core/ | 없음 |
| 2 | Loading | Runtime/Loading/ | Core |
| 3 | Debug | Debug/ | Core |
| 4 | Analysis | Editor/Analysis/ | 없음 |

→ 구현 시작: /ft:build Runtime/Core
```

## 기획 문서 ↔ Feature.md 변환 로직

기획 문서(TASK.md, BRAINSTORM.md)에서 피처를 추출할 때:

1. **표/목록 구조 탐색** — "기능 목록", "Phase 1", "Pillar" 등 패턴
2. **그룹핑 힌트 활용** — 이미 Week/Phase/Pillar로 묶여 있으면 그대로 활용
3. **의존성 키워드** — "~위에 구축", "~기반", "~필요" → 의존성 관계 추출
4. **크기 조절** — 너무 작은 항목은 합치고, 너무 큰 항목은 쪼개기 제안

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 기획 문서 없음 | 사용자와 대화로 직접 요구사항 수집 → 분류 제안 |
| 기획 문서가 너무 많음 | 가장 최신/관련성 높은 것 우선 (수정일 기준) |
| 피처가 1개뿐 | "1개면 /ft:plan이 적합합니다" 안내 후 위임 |
| 피처가 10개 이상 | Phase/단계로 묶어서 "Phase 1에서 N개, Phase 2에서 M개" 제안 |
| 이미 Feature.md가 있는 폴더 | 기존 것 유지, 충돌 시 "업데이트하시겠습니까?" 질문 |
| 사용자가 분류를 크게 변경 | 처음부터 다시 제안하지 않고, 변경 부분만 반영 |

## 경계

**한다:**
- 기획 문서 자동 수집 + 분석
- 피처 분류 제안 + 사용자 확인
- Feature.md × N개 일괄 생성
- 피처 간 의존성 + 구현 순서 도출
- FEATURE_INDEX.md 갱신

**안한다:**
- 코드 작성 (그건 /ft:build 역할)
- 폴더/파일 생성 (Feature.md 외)
- 기존 코드 수정
- 개별 피처 상세 설계 (그건 /ft:plan 역할)
- 사용자 확인 없이 Feature.md 생성
