# /ft:design — 설계 대화

사용자와 대화하며 기술적 결정사항을 하나씩 확정하고, 최종 정리를 출력한다.

## 왜 이 단계가 필요한가

"뭘 만들지"는 알지만 "어떻게 만들지"는 모를 때가 있다.
이 상태에서 바로 /ft:plan을 하면 Feature.md가 모호해지고,
/ft:build를 하면 Claude가 임의로 결정해서 사용자가 원하는 방향과 어긋난다.
설계 대화를 먼저 하면 기술적 결정이 명확해진 상태에서 기획과 구현에 들어갈 수 있다.

## 실행 흐름

### 0단계: 주제 파악

사용자의 요청에서 설계 대상을 파악한다:

- **뭘 만드는지** — "에셋 자동 등록", "인벤토리 시스템" 등
- 주제가 불명확하면 → "어떤 기능의 설계를 하고 싶으신가요?" 질문

### 1단계: 결정 항목 도출

설계 대상에서 결정해야 할 기술적 항목들을 파악한다.
필요하면 프로젝트를 탐색하여 기존 구조와 패턴도 참고한다.

결정 항목 예시:
- 데이터를 어디에 저장할지 (SO, JSON, DB)
- 폴더/파일 구조를 어떻게 잡을지
- 인터페이스를 어떻게 설계할지
- 기존 시스템과 어떻게 연동할지
- 에러/예외를 어떻게 처리할지

항목을 모두 나열하지 않는다. 앞선 결정이 뒤의 선택지에 영향을 주므로 하나씩 진행한다.

### 2단계: 반복 — 하나씩 결정

각 항목에 대해 이 사이클을 반복한다:

```
1. 선택지 제시 (2~3개, 간결하게)
2. 각 선택지의 장단점 (한두 줄)
3. 추천 + 이유 (명확하게)
4. 사용자 선택 대기
5. 확정 후 다음 항목으로
```

#### 대화 규칙

- **질문은 한 번에 하나씩.** 여러 항목을 동시에 묻지 않는다.
- **선택지는 코드블록이나 구조도로 시각화.** 글로만 설명하지 않는다.
- **추천은 반드시 제시.** 사용자가 안 물어도 "X를 추천합니다" + 이유.
- **사용자가 "추천해줘"라고 하면** 추천만 말하고 다른 선택지 상세 설명은 생략.
- **짧은 답변 허용.** "응", "A", "그래", "그걸로" 등 — 이해하고 다음으로 진행.
- **사용자가 역질문하면** 성실하게 답변 후 원래 결정으로 복귀.
- **이전 결정이 다음에 영향을 주면** 그 맥락을 반영하여 선택지 조정.

#### 선택지 제시 형식

```
방식 A: {이름}
  {구조도 또는 코드 예시}
  → 장점: ...
  → 단점: ...

방식 B: {이름}
  {구조도 또는 코드 예시}
  → 장점: ...
  → 단점: ...

추천: A. 이유: ...
```

### 3단계: 전체 정리

모든 결정이 끝나면 확정사항을 깔끔하게 정리하여 출력한다:

```
## {설계 대상} — 최종 설계

### 확정사항

| 항목 | 결정 |
|------|------|
| ... | ... |

### 구조 (해당되면)

{코드블록으로 폴더/파일 구조}

### 다음 단계
→ `/ft:plan`으로 Feature.md 생성
→ `/ft:build`로 구현
```

## 결정 항목이 추가로 필요할 때

대화 중 사용자의 선택에 의해 새로운 결정 항목이 생길 수 있다.
예: "폴더 기반으로 하겠다" → "그러면 폴더 깊이는?" 이 추가 질문이 자연스럽게 발생.

이런 경우 미리 전체 항목 수를 고정하지 말고, 흐름에 따라 질문을 이어간다.
단, 핵심 결정이 끝났는데 지엽적인 것까지 물어보지는 않는다.
"이 정도면 설계가 잡혔다"고 판단되면 3단계(전체 정리)로 넘어간다.

## 프로젝트 분석

필요한 경우 프로젝트를 탐색하여 결정에 참고한다:
- 기존 Feature.md 읽기 (유사 피처 참고)
- 프로젝트 구조/패턴 확인
- 기존 코드의 관례 파악

단, 분석 결과를 장황하게 설명하지 않는다.
선택지에 자연스럽게 녹여서 "현재 프로젝트에서는 X 패턴을 쓰고 있으니 A가 자연스럽습니다" 정도로.

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 사용자가 주제 없이 호출 | "어떤 기능의 설계를 하고 싶으신가요?" 질문 |
| 결정 중 이전 결정 번복 | 해당 항목부터 다시 진행, 영향받는 후속 결정도 재확인 |
| 사용자가 "전부 추천대로" | 모든 추천안으로 확정 후 3단계 정리 |
| 이미 설계가 명확함 | 불필요한 질문 없이 빠르게 정리만 출력 |
| 결정 항목이 1~2개뿐 | 짧게 끝내고 정리. 억지로 늘리지 않음 |

## 경계

**한다:**
- 기술적 결정사항 대화
- 선택지 제시 + 추천 + 이유
- 프로젝트 분석 (참고용)
- 최종 설계 정리 출력

**안한다:**
- 코드 작성 (그건 /ft:build)
- Feature.md 생성 (그건 /ft:plan)
- 파일/폴더 생성
- 커밋/푸시
