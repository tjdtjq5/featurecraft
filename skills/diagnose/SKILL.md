# /ft:diagnose — 근본 원인 진단

버그나 이상 동작의 증상에서 출발하여, 직접 원인과 구조적 원인을 추적하고, 증거 기반으로 해결 옵션을 비교 제시한다.

## 왜 이 스킬이 필요한가

`ft:review`는 코드를 보고 문제를 찾아 바로 고친다. 하지만 어떤 버그는 표면적인 수정만으로는 해결되지 않는다:

- **빠른 패치가 다른 곳을 깨뜨린다** — 원인을 모르고 증상만 막으면 부작용이 생긴다.
- **같은 버그가 반복된다** — 구조적 문제를 안 고치면 다른 곳에서 같은 실수가 재발한다.
- **수정 방향이 여러 개다** — 빠른 수정 vs 구조적 수정, 어느 쪽이 맞는지 판단이 필요하다.

diagnose는 "고치기 전에 왜 일어났는지 먼저 이해한다"는 원칙으로 동작한다.

## ft:review와의 관계

| | ft:review | ft:diagnose |
|---|---|---|
| 입력 | 변경된 코드 (diff) | 버그 증상 / 이상 동작 |
| 시점 | 코드 작성 후 | 수정 방향 결정 전 |
| 출력 | 이슈 리스트 + 패치 | 원인 분석 리포트 + 해결 옵션 |
| 코드 수정 | --fix로 자동 수정 | 안 함 (방향만 제시) |

**워크플로우**: `ft:diagnose` → 방향 결정 → `ft:build` 또는 직접 수정 → `ft:review`

## 실행 흐름

### 1단계: 증상 구조화 (입력)

사용자의 보고에서 다음 4가지를 파악한다:

| 항목 | 파악 대상 | 예시 |
|------|-----------|------|
| **무엇이** | 잘못되는 현상 | "카드 진화 시 강화 수치가 누락된다" |
| **어디서** | 관찰 위치 (코드/화면) | "진화 결과 화면에서 +2가 +3이어야 하는데" |
| **언제** | 발생 조건 | "강화된 카드를 진화할 때만" |
| **기대값** | 정상이라면 어떨지 | "+2 카드 진화 → +3이 정상" |

#### 모호한 입력 처리

4가지 중 핵심 항목이 불분명하면 **분석을 시작하기 전에** 사용자에게 질문한다.

```
"무엇이" 불분명:
  → "구체적으로 어떤 동작이 잘못되나요? (크래시 / 잘못된 결과 / 성능 / 기타)"

"어디서" 불분명:
  → "이 문제가 어느 화면(또는 어느 코드)에서 관찰되나요?"

"언제" 불분명:
  → "항상 발생하나요, 특정 조건에서만 발생하나요?"
```

**질문은 한 번에 하나씩.** 4개를 동시에 묻지 않는다.
단, 사용자 보고가 이미 충분히 구체적이면 질문 없이 바로 2단계로 진행한다.

#### 증상 유형 분류

파악한 정보로 증상을 분류한다 (추적 전략이 달라짐):

| 유형 | 추적 전략 |
|------|-----------|
| **잘못된 결과** (값이 틀림) | 데이터 흐름 역추적 — 값이 어디서 계산/변환되는지 |
| **크래시/예외** | 콜스택 역추적 — null/범위초과/타입 불일치 원점 |
| **동작 안 함** (실행이 안 됨) | 조건 분기 추적 — 어디서 흐름이 끊기는지 |
| **타이밍/순서** (간헐적) | 상태 의존 추적 — 어떤 상태일 때만 발생하는지 |
| **성능** (느림/프리징) | 핫스팟 추적 — 어디서 시간/메모리를 소비하는지 |

### 2단계: 3계층 원인 추적

증상에서 출발하여 3단계 깊이로 원인을 추적한다.

```
[증상] 카드 진화 시 +2 → +2 (정상은 +3)
  ↓
[직접 원인] CardEvolveCase.TryEvolve:31에서 원본 Enhance만 참조
  ↓
[구조적 원인] 진화/강화 계산이 두 곳에 분산, 동기화 지점 없음
```

#### 2-A. 직접 원인 (What)

**어떤 코드가 문제인가** — 기계적 추적.

추적 방법 (증상 유형별):
- **잘못된 결과**: 결과값을 생성하는 코드를 찾고, 입력→출력 데이터 흐름 역추적
- **크래시**: 에러 위치에서 콜체인 역추적, null이 되는 경로 탐색
- **동작 안 함**: 실행 경로의 조건문을 하나씩 확인, 어디서 분기가 빠지는지
- **타이밍**: 상태 변경 시점과 참조 시점의 순서 관계 확인

사용하는 도구:
- `Grep` — 관련 메서드/필드 검색
- `Read` — 의심 코드 읽기
- `feature-explore` 에이전트 — 콜체인이 3개 파일 이상에 걸칠 때

**출력**: 문제가 되는 정확한 코드 위치 (`파일:라인`) + 왜 그 코드가 잘못된 결과를 만드는지.

#### 2-B. 구조적 원인 (Why)

**왜 그런 코드가 존재할 수밖에 없었는가** — 이것이 diagnose의 핵심 가치.

분석 관점:

| 관점 | 질문 | 도구 |
|------|------|------|
| **설계 의도** | 원래 이 코드는 어떤 의도로 만들어졌나? | Feature.md 읽기 |
| **변경 이력** | 문제 코드가 언제/왜 도입됐나? (핫픽스? 원래 설계?) | `git log -p --follow`, `git blame` |
| **패턴 반복** | 같은 실수가 코드베이스에 몇 곳 더 있나? | `Grep`으로 유사 패턴 검색 |
| **책임 분산** | 하나의 책임이 여러 곳에 흩어져 있나? | 관련 클래스/메서드 간 역할 비교 |
| **의존 구조** | 잘못된 의존 방향이나 숨은 커플링이 있나? | using/참조 관계 추적 |

**핵심 판단 기준**:
- 같은 패턴의 실수가 **1곳**이면 → 단순 실수 (빠른 수정 OK)
- 같은 패턴의 실수가 **3곳 이상**이면 → 구조적 문제 (설계 수정 필요)
- 문제 코드가 **핫픽스로 도입**됐으면 → 임시 코드가 영구화된 것 (정리 필요)
- Feature.md에 **해당 동작이 명시 안 됨**이면 → 설계 누락 (Feature.md 갱신 필요)

#### 2-C. 영향 범위

구조적 원인을 수정할 경우 영향받는 코드를 파악한다:

- 문제 코드를 참조하는 곳 (`Grep`으로 호출처 검색)
- 같은 패턴을 공유하는 다른 코드
- 해당 피처에 의존하는 다른 피처 (Feature.md 의존성 역추적)

### 3단계: 증거 검증 (추측 금지)

**모든 주장에 코드 증거를 달아야 한다.** 증거 없는 원인 분석은 출력하지 않는다.

#### 필수 증거 기준

| 주장 유형 | 필수 증거 |
|-----------|-----------|
| "이 코드가 직접 원인이다" | `파일:라인` + 해당 코드가 잘못된 결과를 만드는 구체적 경로 |
| "구조적으로 문제다" | 같은 패턴이 반복되는 위치 목록 (`파일:라인` × N) |
| "이 코드는 핫픽스로 들어왔다" | `git log` / `git blame` 결과 (커밋 해시 + 날짜 + 메시지) |
| "설계 의도와 다르다" | Feature.md의 해당 섹션 인용 vs 실제 코드 비교 |
| "이 수정이 다른 곳에 영향을 준다" | 영향받는 호출처 목록 (`Grep` 결과) |

#### 검증 자가 체크

리포트 출력 전 각 항목에 대해 자가 검증:

```
□ 직접 원인에 파일:라인이 있는가?
□ 구조적 원인에 코드 증거가 있는가? (감상이 아닌 사실)
□ "~일 수 있다", "아마"가 없는가? (있으면 증거 추가 또는 삭제)
□ 영향 범위가 실제 Grep 결과에 기반하는가?
□ 해결 옵션이 현재 코드 구조에서 실행 가능한가?
```

### 4단계: 리포트 출력

```
## 진단 결과: {증상 한 줄 요약}

### 증상
{무엇이 / 어디서 / 언제 / 기대값}

### 직접 원인
**{파일:라인}** — {한 줄 설명}

{코드 블록 + 왜 이 코드가 문제인지 설명}

### 구조적 원인
{왜 그런 코드가 존재하게 됐는지}

- 증거 1: {파일:라인} — {설명}
- 증거 2: {파일:라인} — {설명}
- (패턴 반복인 경우) 동일 패턴 발견: {N}곳
  - {파일1:라인}
  - {파일2:라인}
  - ...

### 영향 범위
- 직접 영향: {N}개 파일 ({파일 목록})
- 간접 영향: {의존 피처 목록, 있으면}

### 해결 옵션

| | A. 빠른 수정 | B. 구조적 수정 |
|---|---|---|
| 내용 | {구체적 변경 내용} | {구체적 변경 내용} |
| 수정 범위 | {N}파일 | {N}파일 |
| 리스크 | {구체적 리스크} | {구체적 리스크} |
| 재발 방지 | {동일 버그 재발 가능성} | {동일 버그 재발 가능성} |
| 추천 조건 | {이 옵션이 적합한 상황} | {이 옵션이 적합한 상황} |

### 추천
{어떤 옵션을 추천하는지 + 이유 (1~2문장)}
```

#### 출력 규칙

- **옵션은 최소 2개, 최대 3개.** 1개면 진단이 아니라 지시. 4개 이상이면 선택 부담.
- **옵션 A는 항상 "가장 작은 변경".** 사용자가 빠르게 해결하고 싶을 때를 위해.
- **각 옵션에 리스크를 반드시 명시.** "리스크 없음"도 OK, 빈칸은 안 됨.
- **추천은 반드시 제시.** 사용자가 안 물어도 추천 + 이유를 달아야 한다.
- **구조적 수정이 과도하면 솔직하게.** "이 버그는 단발성이라 빠른 수정으로 충분합니다"도 유효한 결론.

#### 구조적 원인이 없는 경우

모든 버그가 구조적 문제인 건 아니다. 단순 실수(오타, 로직 착오)일 때:

```
### 구조적 원인
단순 구현 실수. 같은 패턴의 반복 없음 (Grep 확인: 0곳).
구조 변경 불필요.

### 해결 옵션
| | A. 수정 |
|---|---|
| 내용 | {구체적 수정} |
| 수정 범위 | 1파일 |
| 리스크 | 없음 |
```

옵션이 1개뿐이면 비교표 대신 단순 제안으로 줄인다.

## 피처 컨텍스트 활용

진단 대상이 피처 폴더에 있으면:

1. **Feature.md 읽기** — 설계 의도와 현재 코드의 괴리 확인
2. **의존 피처 1단계 체이닝** — 의존 피처의 API 계약이 원인일 수 있음
3. **learnings 확인** — `.featurecraft/learnings/`에 관련 이력이 있는지

Feature.md가 없으면 → 피처 컨텍스트 단계 건너뛰기

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 증상이 너무 모호 ("안돼") | 1단계에서 구체화 질문. 분석 시작 안 함 |
| 여러 버그가 동시에 보고됨 | 하나씩 진단. "어떤 것부터 볼까요?" 질문 |
| 직접 원인은 찾았는데 구조적 원인 판단이 어려움 | 직접 원인만 리포트 + "구조적 원인 불분명" 명시 (추측 금지) |
| 버그가 아니라 의도된 동작 | "이것은 의도된 동작입니다" + 근거 (Feature.md 또는 코드 주석) |
| 재현 불가 (코드에서 원인을 못 찾음) | "코드 분석으로는 원인 특정 불가" + 확인해볼 런타임 조건 제안 |
| 구조적 수정 범위가 너무 큼 (10+ 파일) | 단계적 수정 로드맵 제시 (Phase 1/2/3) |
| 이미 ft:review로 패치한 후 재발 | 이전 패치가 왜 불충분했는지를 구조적 원인에 포함 |

## 경계

**한다:**
- 증상에서 출발한 3계층 원인 추적 (증상 → 직접 → 구조)
- 증거 기반 분석 (파일:라인, git history, 패턴 반복 횟수)
- 해결 옵션 비교표 (빠른 수정 vs 구조적 수정)
- 영향 범위 파악
- Feature.md / git blame 활용
- 추천 + 이유 제시

**안한다:**
- 코드 수정 (방향만 제시, 실행은 사용자가 결정)
- 증거 없는 추측 ("아마 ~일 것이다")
- 변경되지 않은 코드의 일반적 품질 지적 (그건 ft:review)
- 테스트 작성/실행
- Feature.md 직접 수정
