# 프로젝트 스킬 스캔 & 자동 생성

프로젝트를 분석하여 배경지식 스킬이 필요한 영역을 찾고, 승인된 후보를 자동 생성한다.

`ft:skill-creator-project`가 "이 패키지를 스킬로 만들어줘"라면,
이 스킬은 **"프로젝트 전체에서 뭘 스킬로 만들면 좋겠는지 찾아줘"** 이다.

## 왜 이 스킬이 필요한가

프로젝트가 커지면 커스텀 패키지, 아키텍처 패턴, 프레임워크 규칙이 늘어난다.
Claude가 이런 프로젝트 고유 지식 없이 코드를 작성하면 잘못된 API를 사용하거나 금지된 패턴을 쓴다.
배경지식 스킬이 이를 방지하지만, "어디에 스킬이 필요한지" 판단 자체가 어렵다.

## 실행 흐름

### Step 1: 프로젝트 인벤토리 수집

세 가지 소스에서 병렬로 정보를 수집한다:

**1-A: 피처 맵 (Feature.md 기반)**

```
Glob: **/Feature.md
```

각 Feature.md에서 추출:
- 피처명, 경로, 한 줄 설명
- 주요 클래스/시스템 목록
- 의존성 (다른 피처/패키지 참조)

Feature.md가 없는 피처 폴더도 감지한다 (Glob으로 `_Feature/*/` 탐색).

**1-B: 커스텀 패키지 목록**

```
Read: Packages/manifest.json
```

`com.{사용자}.*` 패턴의 커스텀 패키지를 추출한다.
각 패키지의 package.json에서 이름, 버전, 설명을 읽는다.

**1-C: 기존 스킬 목록**

```
Glob: .claude/skills/*/SKILL.md
```

각 스킬의 frontmatter에서 name, description을 읽는다.
description의 트리거 키워드 목록을 파싱한다.

### Step 2: 갭 분석

수집된 데이터를 대조하여 커버리지를 판단한다.

**스킬이 커버하는 영역:**
기존 스킬의 description 키워드 + 본문의 피처 맵을 기준으로,
어떤 패키지/피처/패턴이 이미 스킬로 커버되는지 매핑한다.

**스킬 후보 판정 기준:**

"스킬로 만들 가치가 있는가?"를 아래 기준으로 판단한다:

| 기준 | 높음 (스킬 필요) | 낮음 (불필요) |
|------|-----------------|--------------|
| **고유성** | 커스텀 패키지, 프로젝트 고유 패턴 | 표준 Unity API, 공개 라이브러리 |
| **복잡도** | 모듈 3개+, 파이프라인 있음, 규칙 위반 시 버그 | 파일 1~2개, 단순 유틸리티 |
| **사용 빈도** | 여러 피처가 의존 | 한 피처 내부에서만 사용 |
| **실수 가능성** | 금지 패턴 존재, 특정 순서 필수 | 직관적인 API |

**후보 카테고리:**

1. **패키지 스킬** — 커스텀 UPM 패키지 (고유 API, Claude가 모름)
2. **아키텍처 스킬** — 크로스커팅 패턴 (DI 규칙, ECS↔MB 브릿지 등)
3. **프레임워크 스킬** — 외부 프레임워크 + 프로젝트 고유 사용법 (Spine, DOTween 등)

피처 단위 스킬은 일반적으로 **불필요** — Feature.md가 이미 그 역할을 한다.
단, 피처가 다른 피처들의 기반 역할을 하고 규칙이 복잡하면 예외.

### Step 3: 리포트 출력

아래 형식으로 스캔 결과를 보여준다:

```markdown
## 스킬 스캔 결과

### 기존 스킬 커버리지
| # | 스킬 | 대상 | 트리거 키워드 수 | 상태 |
|---|------|------|----------------|------|
| - | ecs-guide | Unity ECS | 14개 | ✓ |
| - | addrx-guide | Addressables | 11개 | ✓ |

### 스킬 후보
| # | 후보명 | 대상 | 카테고리 | 추천 이유 |
|---|--------|------|----------|-----------|
| 1 | {name} | {대상} | 패키지 | {이유} |
| 2 | {name} | {대상} | 아키텍처 | {이유} |

### 커버 불필요 (이유 포함)
| 대상 | 이유 |
|------|------|
| {피처X} | Feature.md로 충분 |
| {라이브러리Y} | 표준 API, 공개 문서 풍부 |
```

### Step 4: 사용자 선택

> 생성할 스킬을 선택하세요 (번호, 쉼표 구분, 또는 'all'):

사용자가 선택하면 각 후보에 대해:
1. 트리거 키워드 제안을 보여주고 확인받는다
2. 승인된 후보를 생성한다

### Step 5: 스킬 자동 생성

승인된 각 후보에 대해 `/ft:skill-creator-project {대상 경로}` 의 템플릿을 따라 생성한다.

생성 위치: `.claude/skills/{스킬명}/SKILL.md`

**트리거 키워드 추출 전략:**

좋은 키워드 = "이 키워드가 대화에 나오면 이 스킬이 필요할 확률이 높다"

추출 소스:
1. **클래스/인터페이스명** — public 클래스 중 자주 사용되는 것 (예: `StatOwner`, `SkillDef`)
2. **네임스페이스 키워드** — 핵심 네임스페이스 단어 (예: `InGameCore`, `Stats`)
3. **Attribute명** — 프레임워크가 Attribute를 쓰면 반드시 포함 (예: `[Table]`, `[Config]`)
4. **Feature.md의 "용도" 섹션** — 한국어 키워드 추출 (예: '스탯', '버프', '디버프')
5. **금지 패턴의 대안 키워드** — 금지 사항이 있으면 해당 키워드도 (예: `SystemBase` → ecs-guide 발동)

나쁜 키워드 = 너무 일반적 (예: 'player', 'enemy', 'UI') → 오발동 유발

**생성 후 확인:**
```
✓ {N}개 스킬 생성 완료:
  - .claude/skills/{skill-1}/SKILL.md
  - .claude/skills/{skill-2}/SKILL.md

수정이 필요하면 알려주세요.
```

## 업데이트 모드

기존 스킬이 오래되었는지도 체크한다.

기존 스킬의 피처 맵에 나열된 파일이 실제로 존재하는지 Glob으로 확인.
없어진 파일이 있으면 "⚠ 업데이트 필요"로 표시.

사용자가 업데이트를 요청하면:
1. 해당 스킬의 대상 경로를 다시 분석
2. 변경된 부분만 반영하여 SKILL.md 업데이트

## 경계

**한다:**
- Feature.md, packages, 기존 스킬 스캔
- 갭 분석 + 후보 제안
- 승인된 후보의 SKILL.md 자동 생성
- 기존 스킬 유효성 체크

**안한다:**
- 피처 단위 스킬 생성 (Feature.md가 그 역할)
- 글로벌 스킬 생성 (~/.claude/skills/)
- 코드 구현 (스킬 생성만)
