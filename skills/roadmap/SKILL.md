# /ft:roadmap — 로드맵 기획

프로젝트/패키지 단위의 큰 그림을 잡고,
Pillar/모듈별 Feature.md를 일괄 생성한다.

## 왜 이 스킬이 필요한가

Feature.md 없이 /ft:build를 할 수 없다.
새 피처 1개든, 대규모 로드맵이든 먼저 Feature.md를 만들어야 한다.

/ft:roadmap이 이 역할을 통합한다:
- TASK.md, BRAINSTORM.md 같은 기획 문서를 읽고
- 피처 단위로 쪼개서 Feature.md를 생성하고 (1개든 N개든)
- 의존성 기반 구현 순서를 제안한다.

## 실행 흐름

### 0단계: 모호성 체크

사용자의 요청에서 아래 3가지가 파악되는지 확인한다:

1. **목표** — 뭘 만드는 프로젝트/패키지인지 (예: "Addressables 확장 패키지")
2. **범위** — 어디까지 만드는지 (예: "런타임 안전 + 디버깅 + 검증")
3. **성공 기준** — MVP 완성의 정의 (예: "SafeHandle로 로드/해제 + 누수 감지 동작")

3가지 중 하나라도 빠지면:
- 빠진 항목에 대해 질문한다
- 질문은 한 번에 하나씩, 목표부터
- 3가지가 모두 파악되면 1단계로 진행

### 1단계: 기획 문서 수집

**프로젝트 루트부터** 기획 관련 문서를 탐색한다.
대상 패키지 폴더만 보면 안 된다 — 기획 문서는 별도 폴더에 있는 경우가 많다.
(예: `Assets/Address/TASK.md`가 `Assets/AddrX/` 패키지의 기획 문서일 수 있다)

#### 탐색 순서

1. **프로젝트 루트에서 Glob 검색** (대상 패키지에 한정하지 않는다):
   ```
   **/TASK.md
   **/BRAINSTORM.md
   **/RESEARCH.md
   **/README.md
   **/Feature.md
   .featurecraft/FEATURE_INDEX.md
   ```

2. **관련 폴더 탐색** — 대상 패키지와 이름이 유사한 폴더가 있으면 반드시 확인:
   - 예: 대상이 `AddrX`이면 → `Address`, `Addressable`, `Addr` 등도 탐색
   - 예: 대상이 `SupaRun`이면 → `GameServer`, `Server` 등도 탐색
   - 이런 폴더에 TASK.md, BRAINSTORM.md 등이 있으면 기획 문서로 간주

3. **발견된 문서를 읽고 핵심 정보를 추출한다**:
   - 대상 패키지와 관련 있는 내용만 필터링
   - 각 문서에서 피처 목록, 의존성, 우선순위 정보 추출

- 문서가 하나도 없으면 → 사용자에게 직접 요구사항을 질문한다
- 문서가 있으면 → "이 문서들을 기반으로 로드맵을 잡겠습니다" 안내

### 2단계: 피처 분류 제안

기획 문서 + 사용자 요구사항을 기반으로 피처 묶음을 제안한다.

#### 적정 피처 크기 판단 기준

| 기준 | 너무 작음 | 적정 | 너무 큼 |
|------|-----------|------|---------|
| 파일 수 (예상) | 1~2개 | 3~10개 | 20개+ |
| 독립 이해 가능 | 다른 것과 합쳐야 이해됨 | 단독으로 이해/구현 가능 | 여러 책임 혼재 |
| 구현 단위 | 함수 하나 | 모듈/폴더 하나 | 프로젝트 절반 |

#### 제안 포맷

```
## 로드맵 제안: {프로젝트명}

### Pillar 1: {이름} — {한줄 설명}
포함 기능: A, B, C
의존성: 없음 (기반)
예상 위치: {경로}

### Pillar 2: {이름} — {한줄 설명}
포함 기능: D, E, F
의존성: Pillar 1
예상 위치: {경로}

...

### 구현 순서
1. Pillar 1 (기반, 의존성 없음)
2. Pillar 2 (Pillar 1 의존)
3. Pillar 3 (독립)
4. Pillar 4 (Pillar 1, 3 의존)
```

### 3단계: 사용자 확인

- 제안된 피처 분류를 사용자에게 보여준다
- 피드백을 받아 조정한다:
  - 피처 합치기 / 쪼개기
  - 이름 변경
  - 포함 기능 이동
  - 구현 순서 변경
  - 범위 축소/확장
- **사용자가 "좋아" / 확인하면 4단계로 진행**

### 4단계: Feature.md 일괄 생성

확정된 피처 분류에 따라 Feature.md를 한번에 생성한다.
(하나씩 확인받지 않는다 — 3단계에서 이미 확인받았으므로)

각 Feature.md 포맷:

```markdown
# {피처명}

## 상태
wip

## 용도
{한 줄 설명}

## 의존성
- {상대경로} — {참조 이유}

## 포함 기능
- {기능 1} — {설명}
- {기능 2} — {설명}
- ...

## 구조
(아직 없음 — /ft:build 후 업데이트)

## API (외부 피처가 참조 가능)
(아직 없음 — /ft:build 후 업데이트)

## 주의사항
- {있으면 기재}
```

`## 포함 기능` 섹션은 이 피처 안에 들어갈 세부 기능 목록이다.
/ft:build가 이 목록을 보고 구현 범위를 파악한다.

### 5단계: tasks.md 일괄 생성 (각 피처마다)

각 Feature.md를 생성한 직후, **항상** `.featurecraft/tasks/{feature-kebab}-tasks.md`를 함께 생성한다.
(태스크가 1개뿐이어도 생성 — 일관성 유지)

**목적**:
- `/ft:build`가 체크박스를 읽어 태스크 단위로 진행을 추적한다
- 사용자가 난이도/분량을 보고 구현 계획을 조정할 수 있다
- Phase별로 작업량을 가늠할 수 있다

#### 태스크 쪼개기 기준

`## 포함 기능` 섹션을 보고 **구현 태스크로 잘게 쪼갠다**:

| 기준 | 설명 |
|------|------|
| **독립 실행 가능** | 태스크 하나가 독립적으로 완결되어야 함 (중간에 깨진 상태 X) |
| **파일/커밋 단위** | 보통 1~3파일 변경 또는 1커밋 분량 |
| **최소 15분 이상** | 너무 잘게 쪼개면 관리 비용이 커짐 |
| **Phase 구분** | 의존 관계가 있는 태스크는 Phase로 묶음 (Phase 1 → Phase 2) |

#### 난이도/분량 메타 작성 기준

난이도:
- `쉬움`: 기존 패턴 복제, 단순 추가, 값 변경
- `보통`: 새 로직, 여러 파일 연동, 인터페이스 정의
- `어려움`: 아키텍처 변경, 복잡한 통합, 여러 시스템 영향

분량:
- `1파일`, `3파일` 등 — 생성/수정할 파일 수
- `5줄 수정`, `50줄 추가` 등 — 변경 라인이 핵심일 때
- 둘 다 해당하면 우세한 기준으로 표시

roadmap이 작성하는 메타는 **초안**이므로, 모든 태스크에 `(초안)` 표시를 붙인다.
사용자가 조정하면 `(초안)` 표시를 제거한다.

#### tasks.md 포맷

```markdown
# {피처명} 구현 태스크

> `/ft:build`가 이 파일의 체크박스를 읽어 진행을 추적합니다.
> 태스크 수정/추가/삭제는 직접 편집 가능합니다.
> 난이도/분량은 roadmap이 작성한 초안입니다 — 조정 후 `(초안)` 표시를 제거하세요.

## Phase 1: 기반 (예상: ~2시간)

- [ ] 인터페이스 정의 (IXxxService.cs)  `<난이도: 보통, 분량: 1파일>` (초안)
- [ ] Mock 구현 (MockXxxService.cs)  `<난이도: 쉬움, 분량: 1파일>` (초안)
- [ ] ServiceLocator 등록  `<난이도: 쉬움, 분량: 2줄 수정>` (초안)

## Phase 2: 통합 (예상: ~3시간)

- [ ] 호출부 리팩터링  `<난이도: 어려움, 분량: 5파일>` (초안)
- [ ] 테스트 작성  `<난이도: 보통, 분량: 2파일>` (초안)
```

**Phase 없이 단일 리스트로 충분한 소규모 피처**는 `## 태스크` 한 섹션으로 작성해도 된다.

### 6단계: 마무리

1. **FEATURE_INDEX.md 갱신** — 생성된 모든 피처 등록
2. **구현 순서 안내** — "다음 단계: `/ft:build {첫 번째 피처 경로}`로 구현을 시작하세요."
3. **생성 결과 요약** 출력:

```
## 로드맵 생성 완료

| # | 피처 | Feature.md | 태스크 | 의존성 |
|---|------|------------|--------|--------|
| 1 | Core | Runtime/Core/Feature.md | .featurecraft/tasks/core-tasks.md | 없음 |
| 2 | Loading | Runtime/Loading/Feature.md | .featurecraft/tasks/loading-tasks.md | Core |
| 3 | Debug | Debug/Feature.md | .featurecraft/tasks/debug-tasks.md | Core |
| 4 | Analysis | Editor/Analysis/Feature.md | .featurecraft/tasks/analysis-tasks.md | 없음 |

→ 구현 시작: /ft:build Runtime/Core
```

## 기획 문서 ↔ Feature.md 변환 로직

기획 문서(TASK.md, BRAINSTORM.md)에서 피처를 추출할 때:

1. **표/목록 구조 탐색** — "기능 목록", "Phase 1", "Pillar" 등 패턴
2. **그룹핑 힌트 활용** — 이미 Week/Phase/Pillar로 묶여 있으면 그대로 활용
3. **의존성 키워드** — "~위에 구축", "~기반", "~필요" → 의존성 관계 추출
4. **크기 조절** — 너무 작은 항목은 합치고, 너무 큰 항목은 쪼개기 제안

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 기획 문서 없음 | 사용자와 대화로 직접 요구사항 수집 → 분류 제안 |
| 기획 문서가 너무 많음 | 가장 최신/관련성 높은 것 우선 (수정일 기준) |
| 피처가 1개뿐 | 1개짜리 Feature.md + tasks.md를 바로 생성 (Pillar 분류 생략) |
| 피처가 10개 이상 | Phase/단계로 묶어서 "Phase 1에서 N개, Phase 2에서 M개" 제안 |
| 이미 Feature.md가 있는 폴더 | 기존 것 유지, 충돌 시 "업데이트하시겠습니까?" 질문 |
| 이미 tasks.md가 있음 | 기존 체크박스 상태 보존, 새 태스크만 추가 제안 (덮어쓰지 않음) |
| 사용자가 분류를 크게 변경 | 처음부터 다시 제안하지 않고, 변경 부분만 반영 |
| 태스크가 1개뿐인 피처 | Phase 구분 없이 `## 태스크` 단일 섹션으로 생성 |

## 경계

**한다:**
- 기획 문서 자동 수집 + 분석
- 피처 분류 제안 + 사용자 확인
- Feature.md × N개 일괄 생성
- **tasks.md × N개 일괄 생성** (체크박스 + 난이도/분량 초안)
- 피처 간 의존성 + 구현 순서 도출
- FEATURE_INDEX.md 갱신

**안한다:**
- 코드 작성 (그건 /ft:build 역할)
- 폴더/파일 생성 (Feature.md 외)
- 기존 코드 수정
- 개별 피처 상세 설계 (그건 /ft:design 역할)
- 사용자 확인 없이 Feature.md 생성
