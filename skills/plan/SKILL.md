# /ft:plan — 피처 기획

사용자와 대화하며 요구사항을 정리하고, 피처 폴더에 Feature.md를 생성한다.

## 왜 이 단계가 필요한가

Claude는 기존에 만들어진 기능을 참고하지 않고 자기 방식으로 새로 만드는 경향이 있다.
Feature.md는 "이 피처가 뭘 하고, 뭘 의존하고, 어디에 뭐가 있는지" 알려주는 지도다.
이 지도가 없으면 /ft:build가 기존 코드를 무시하고 처음부터 새로 만들어버린다.

## 실행 흐름

### 0단계: 모호성 체크 (Feature.md 생성 전 반드시 통과)

사용자의 요청에서 아래 3가지가 파악되는지 확인한다:

1. **목표** — 뭘 만드는지 (예: "인벤토리 시스템", "버프 기능")
2. **범위** — 어디까지 만드는지 (예: "아이템 추가/삭제/조회")
3. **성공 기준** — 뭐가 되면 완성인지 (예: "상점에서 아이템 구매 시 인벤토리에 추가")

3가지 중 하나라도 빠지면:
- Feature.md를 생성하지 않는다
- 빠진 항목에 대해 질문한다
- 질문은 한 번에 하나씩, 가장 근본적인 것(목표)부터
- 3가지가 모두 파악되면 1단계로 진행

이 단계의 목적: 모호한 상태에서 Feature.md를 만들면 방향이 잘못 잡히고,
나중에 전부 다시 만들게 된다. 먼저 명확하게 만드는 게 훨씬 빠르다.

### 1단계: 요구사항 파악
0단계를 통과했으면 사용자에게 추가 질문하며 세부 정보를 수집한다:

- **뭘 만드나?** — 피처의 목적과 핵심 기능
- **어디에 만드나?** — 프로젝트 내 위치 (사용자 지정 또는 자동 제안)
- **뭘 참고하나?** — 기존 피처 중 의존할 것

질문은 한 번에 하나씩. 사용자가 충분히 설명했으면 불필요한 질문을 하지 않는다.

### 2단계: 프로젝트 분석
요구사항이 잡히면 프로젝트를 분석한다:

1. 프로젝트 내 기존 Feature.md 검색 (Glob으로 `**/Feature.md`)
2. 관련 피처가 있으면 Feature.md를 읽어서 API 파악
3. 사용자가 위치를 지정하지 않았으면, 프로젝트 구조를 보고 적절한 위치 제안
   - 예: "InGame 폴더 안에 이미 Inventory, Shop이 있으니, 여기에 만드는 게 자연스러워 보입니다"

### 3단계: Feature.md 생성
아래 포맷으로 Feature.md를 생성한다:

```markdown
# {피처명}

## 상태
wip

## 용도
{한 줄 설명}

## 의존성
- {상대경로} — {참조 이유}

## 구조
(아직 없음 — /ft:build 후 업데이트)

## API (외부 피처가 참조 가능)
(아직 없음 — /ft:build 후 업데이트)

## 주의사항
- {있으면 기재}
```

### 4단계: 확인 + 피처 인덱스 갱신
- 생성된 Feature.md를 사용자에게 보여주고 확인받기
- 수정 요청이 있으면 반영
- `.featurecraft/FEATURE_INDEX.md` 갱신 (없으면 생성)
- 완료 후: "Feature.md가 준비되었습니다. `/ft:build`로 구현을 시작하세요." 안내

## 위치 제안 로직

사용자가 "어디에 만들지" 지정하지 않았을 때:

1. 프로젝트 내 기존 Feature.md들의 위치 패턴 분석
2. 비슷한 성격의 피처가 이미 모여있는 폴더 찾기
3. 후보 경로를 1~2개 제안하고 사용자에게 선택시키기
4. 기존 Feature.md가 하나도 없으면 → 사용자에게 직접 물어보기

## 계층 피처 처리

상위 폴더에 Feature.md가 이미 있으면:
- 새 피처를 하위로 추가
- 상위 Feature.md의 "하위 피처" 섹션에 새 항목 추가

## 의존성 탐색

사용자가 "바인딩 기능 써서" 같이 명시하면:
1. 프로젝트에서 해당 피처의 Feature.md 찾기
2. API 섹션 읽어서 뭘 활용할 수 있는지 파악
3. 의존성에 추가

사용자가 명시하지 않아도:
1. 요구사항 키워드와 기존 Feature.md의 용도 매칭
2. 관련 피처가 있으면: "기존에 {피처명}이 있는데, 이걸 활용하시겠습니까?" 제안

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 같은 이름 피처가 이미 있음 | "이미 {경로}에 존재합니다. 수정하시겠습니까?" |
| 사용자 요구가 너무 모호 | Feature.md 생성 금지. 0단계 모호성 체크에서 질문으로 명확화 |
| 의존할 피처의 Feature.md가 없음 | "/ft:scan으로 먼저 등록하시겠습니까?" 안내 |
| 의존성 순환 감지 | 경고 + 어디서 끊을지 사용자에게 질문 |
| deprecated 피처 의존 | 경고 + 대체 피처 있으면 제안 |

## 경계

**한다:**
- 사용자와 대화하며 요구사항 정리
- 프로젝트 분석 + 기존 Feature.md 탐색
- Feature.md 생성 + 의존성 설정
- 상위 Feature.md 업데이트

**안한다:**
- 코드 작성 (그건 /ft:build 역할)
- 폴더/파일 생성 (Feature.md 외)
- 기존 코드 수정
- 테스트 작성
- 목표/범위/성공기준이 불명확한 상태에서 Feature.md 생성 (반드시 먼저 질문)
