# ft Plugin Usage Rules

이 프로젝트는 ft(featurecraft) 플러그인을 사용 중입니다.
`.featurecraft/FEATURE_INDEX.md`가 존재하며, 아래에 그 내용이 함께 포함되어 있습니다.
코드 탐색·작성·리뷰·설계·기획 작업에서 **ft 에이전트와 스킬을 우선 활용**하세요.

---

## feature-explore — 코드 탐색의 기본값

코드 구조를 파악하거나 여러 파일을 탐색해야 할 때 우선 선택지.

**반드시 사용**:
- 구조 질문: "어떻게 동작해", "흐름 추적", "관련 코드 전부"
- 의존성 추적: "A가 B에 어떻게 연결되는지"
- 여러 피처에 걸친 조사 (크로스 피처)
- **3회 이상 도구 호출이 예상되는 탐색**

**사용 안 해도 됨** (직접 도구가 더 빠름):
- 정확한 경로를 아는 단일 파일 열람 → `Read`
- 특정 심볼 1회 검색 → `Grep`
- 특정 파일 패턴 1회 검색 → `Glob`

---

## feature-writer — 코드 작성의 기본값

새 코드를 작성하거나 기존 코드를 구조적으로 수정해야 할 때 우선 선택지.

**반드시 사용**:
- 새 클래스/파일/피처 추가
- 기존 API 재사용이 중요한 구현 (의존 피처가 있는 경우)
- 책임성 분리·유지보수성 원칙 적용이 필요한 구조적 변경
- 여러 파일을 수정하는 구현

**사용 안 해도 됨** (직접 `Edit`이 더 빠름):
- 단순 메서드 수정, 버그 픽스, 타이포
- 값 변경, 상수 조정
- 1~2줄 변경

---

## ft 스킬·커맨드 매핑

사용자 발화에서 아래 키워드가 나타나면 해당 도구를 사용.
**자동 호출**은 Claude가 직접 발동, **사용자 확인 필요**는 제안 후 승인 대기.

### 자동 호출 (스킬)
| 사용자 발화 키워드 | 호출 |
|-------------------|------|
| "설계하자", "어떻게 만들지", "방향 잡자", "선택지 달라" | `ft:design` 스킬 |
| "로드맵", "큰 그림", "MVP", "피처 기획", "새 피처" | `ft:roadmap` 스킬 |
| "리뷰해줘", "검토해줘", "코드 봐줘", "이거 괜찮아?" | `ft:review` 스킬 |
| "왜 이런 버그가", "원인 분석", "근본 원인", "왜 이렇게 동작해", "진단해줘", "왜 안돼", "왜 이래" | `ft:diagnose` 스킬 |
| "프로젝트 분석", "스캔", "구조 파악", "피처 등록" | `ft:scan` 스킬 |

### 사용자 확인 필요 (Git 슬래시 커맨드)
| 사용자 발화 키워드 | 제안 |
|-------------------|------|
| "커밋하고 머지", "한 번에 반영", "PR 만들어서 머지" | `/ft:push` |
| "기본브랜치 최신", "dev 반영", "최신 받아줘" | `/ft:pull` |
| "릴리스해줘", "main 배포" | `/ft:release` |

### feature-writer vs /ft:build 사용 기준

| 기준 | feature-writer (에이전트) | /ft:build (커맨드) |
|------|--------------------------|-------------------|
| 호출 | Claude 자동 위임 | 사용자 명시 호출만 |
| 규모 | 1~5 파일 | 제한 없음 |
| learnings | 읽기 | 읽기 + 쓰기 |
| Feature.md 검증 | 갱신만 | 코드↔Feature.md 정밀 대조 |
| 리뷰 연계 | 3파일+ 시 제안 | --review로 자동 실행 |
| 가이드 스킬 체크 | 안 함 | 함 |

- "구현해줘", "만들어줘" → `feature-writer`
- `/ft:build` 명시 호출 → `ft:build`
- 새 피처 전체 구현 → `/ft:build` 추천

### 참고
- `/ft:dev`, `/ft:pkg-*`, `/ft:skill-scan`은 플러그인 개발자용이므로 일반 대화에서는 호출하지 않는다.

## Feature.md 기본 의무

- 피처 폴더에서 작업할 때는 **해당 피처의 Feature.md를 먼저 읽는다**.
- 의존 피처의 Feature.md도 **1단계 깊이로 체이닝** (Feature.md의 "의존성" 섹션 참조).
- 코드 변경 후 다음이 바뀌면 **Feature.md를 갱신한다**: 새 public API, 새 파일, 새 의존 피처, 발견한 제약.

---

## ft 도구 vs 직접 도구 판단

- **탐색**: 3회 이상 도구 호출 예상 → `feature-explore`, 1~2회 → 직접 `Read`/`Grep`/`Glob`
- **작성**: 새 클래스/피처/구조 변경 → `feature-writer`, 단순 수정 → 직접 `Edit`

---

## 새 피처 추가 워크플로

`/ft:design` → `/ft:roadmap` → `feature-writer` (또는 `/ft:build`) → `/ft:review --fix` → `/ft:push`
