# /ft:skill-creator-project — 프로젝트 전용 스킬 생성

특정 패키지/모듈을 분석하고, Claude가 해당 코드를 자동으로 참조하도록 강제하는 프로젝트 레벨 스킬을 생성한다.

## 사용법

```
/ft:skill-creator-project {패키지명 또는 폴더 경로}
```

예시:
- `/ft:skill-creator-project tjdtjq5.editortoolkit`
- `/ft:skill-creator-project Assets/Scripts/Core/NetworkManager`
- `/ft:skill-creator-project Packages/com.company.uiframework`

## 실행 흐름

### Step 1: 대상 찾기

$ARGUMENTS로 전달된 패키지명 또는 경로를 프로젝트에서 찾는다:

1. 정확한 경로면 → 바로 사용
2. 패키지명이면 → 아래 순서로 탐색:
   - `Packages/com.{name}/` (Unity 패키지 컨벤션)
   - `Packages/` 하위에서 package.json의 name 매칭
   - `Assets/` 하위에서 asmdef의 name 매칭
   - Glob으로 `**/*{name}*` 패턴 검색
3. 못 찾으면 → "해당 패키지를 찾을 수 없습니다. 정확한 경로를 알려주세요." 안내 후 중단

### Step 2: 코드 분석

**2-A: Feature.md 우선 읽기**

패키지 내 기존 문서를 먼저 활용한다 (탐색 비용 절감):

1. `Feature.md` 파일들을 Glob으로 찾아 모두 읽는다
2. `README.md`, `CHANGELOG.md`가 있으면 읽는다
3. `FEATURE_INDEX.md`가 있으면 읽는다 (피처 전체 목록)

이 문서들에서 파악할 수 있는 것:
- 피처/모듈 단위 구조와 역할
- 주요 클래스와 의존성
- 이미 정리된 API 요약

**2-B: 코드 보충 분석**

Feature.md에서 파악이 안 된 부분만 Explore 에이전트로 분석한다:

- 어셈블리 이름 + 네임스페이스
- public 클래스/struct/enum 중 문서에 없는 것
- 베이스 클래스 (상속 패턴)
- static 유틸리티/헬퍼

**2-C: 데이터 흐름 파이프라인 분석 (핵심)**

패키지 내에서 **데이터가 모듈을 넘나드는 흐름**을 추적한다:

1. 한 모듈의 출력이 다른 모듈의 입력이 되는 관계를 파악
2. 예: Attribute 정의 → Source Generator가 읽음 → 코드 생성 → 템플릿에서 렌더링
3. 각 흐름의 **진입점 파일:라인**을 기록
4. 새 기능 추가 시 어떤 파일들을 함께 수정해야 하는지 파악

파이프라인 예시:
```
[Json] 어트리뷰트 → DefGenerator (SourceGen~) → ServerCodeGenerator (Editor/Deploy) → AdminTemplate (Templates~)
```

**분석 결과를 사용자에게 요약해서 보여준다:**
```
## 분석 결과: {패키지명}

### 피처 구조
| 피처 | 경로 | 역할 |
|------|------|------|
| Auth | Runtime/Auth/ | 인증 + OAuth |
| Client | Runtime/Client/ | HTTP 통신 |
| ... | ... | ... |

### 주요 파이프라인
1. Attribute → SourceGen → ServerCode → AdminUI (4개 모듈 관통)
2. Config 정의 → DefGenerator → Def 클래스 자동 생성
3. ...

### 기본 정보
- 어셈블리: Assembly.Name
- 네임스페이스: Namespace.Name
- 피처: N개, 클래스: M개
```

### Step 3: 사용자와 스킬 범위 확정

사용자에게 질문한다:

> "이 패키지를 어떤 상황에서 강제로 참조하게 할까요?"
>
> 예시:
> 1. 에디터 코드 작성 시
> 2. UI 코드 작성 시
> 3. 네트워크 관련 코드 작성 시
> 4. 모든 코드 작성 시
> 5. (직접 입력)

사용자 답변에 따라:
- description의 트리거 키워드 결정
- 스킬명 자동 제안 (패키지명에서 추출, 사용자 확인)

### Step 4: SKILL.md 직접 생성

Step 1~3에서 수집한 정보로 `.claude/skills/{스킬명}/SKILL.md`를 직접 작성한다.

**사전 체크:**
- 같은 이름의 스킬이 이미 있으면 → "이미 {경로}에 존재합니다. 덮어쓰시겠습니까?" 확인
- `.claude/skills/` 폴더가 없으면 생성
- 프로젝트 레벨에만 생성 (글로벌 ~/.claude/skills/ 에는 절대 생성하지 않음)

**SKILL.md 템플릿:**

```markdown
---
name: "{스킬명}"
description: "{발동 조건을 구체적으로 서술}. {트리거 키워드들을 나열}."
user-invocable: false
---

# {패키지/모듈명} 탐색 가이드

이 프로젝트에는 `{패키지명}` 패키지가 설치되어 있다.
{발동 조건} 코드를 작성할 때 이 패키지의 API를 사용해야 한다.

**패키지 루트:** `{패키지 절대 경로}`

## 왜 이 규칙이 중요한가

{패키지를 무시하고 직접 구현했을 때의 문제점 2~3가지}

## 피처 맵

패키지는 아래 피처/모듈로 구성된다. 작업 대상 피처의 Feature.md를 먼저 읽어라.

| 피처 | 경로 | 역할 | 핵심 파일 |
|------|------|------|-----------|
| {피처A} | {상대경로}/ | {한 줄 설명} | {진입점 파일명} |
| {피처B} | {상대경로}/ | {한 줄 설명} | {진입점 파일명} |
| ... | ... | ... | ... |

모듈을 넘나드는 주요 데이터 흐름. **새 기능 추가 시 관련 파이프라인의 모든 단계를 확인하라.**

| ... | ... | ... |

### 파이프라인 2: {이름}

## 핵심 API 요약

{카테고리별로 분류하여 API 나열}
{각 API: 메서드 시그니처 + 한 줄 설명}
{베이스 클래스가 있으면 상속 예시 코드블록 포함}

## 사용 패턴

{대표적인 사용 시나리오 1~3개를 코드블록으로}

## 금지 사항

{패키지에 대응 API가 있는데 직접 구현하면 안 되는 것들}
{형식: "X 대신 Y 사용" — 구체적인 API명 포함}

패키지에 대응하는 API가 없는 경우에만 기본 API를 직접 사용한다.
확실하지 않으면 패키지 코드를 먼저 읽어본다: `{패키지 경로}`
```

**작성 규칙:**

1. **description이 가장 중요** — 자동 발동의 정확도를 결정한다
   - 발동 조건을 구체적으로 (예: "Unity Editor 코드를 작성하거나 수정할 때")
   - 트리거 키워드를 **한국어 + 영어 모두** 나열 (예: "'에디터', 'Editor', 'Inspector', 'EditorWindow', '인스펙터' 키워드가 나오면")
   - 패키지의 **고유 클래스명/어트리뷰트명**도 키워드에 포함 (예: '[Table]', '[Config]', 'DataRegistry')
2. **피처 맵이 핵심** — "어디를 봐야 하는지"를 즉시 알 수 있게
   - 각 피처의 경로 + 역할 + 핵심 진입점 파일을 테이블로
   - Feature.md가 있는 피처는 "Feature.md를 읽어라" 지시
3. **파이프라인은 크로스커팅 질문의 답** — "A를 추가하면 어디를 고쳐야 하나?"
   - 모듈을 넘나드는 데이터 흐름을 단계별 테이블로
   - 각 단계에 파일 경로 포함
4. **핵심 API는 스킬 본문에 직접 포함** — 매번 패키지를 읽지 않아도 되게
5. **금지 사항은 "X 대신 Y" 형식** — 모호하지 않게
6. **패키지가 큰 경우 (100+ 파일)** — 피처 맵 + 파이프라인은 반드시 포함, API 상세는 "Feature.md 또는 코드를 직접 읽어라" 지시

### Step 5: 완료 안내

```
✓ 스킬 생성 완료: .claude/skills/{스킬명}/SKILL.md

이제 {발동 조건} 시 자동으로 {패키지명}을 참조합니다.
패키지 코드가 크게 변경되면 이 스킬도 업데이트가 필요합니다.

수정할 부분이 있으면 알려주세요.
```

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| 인자 없음 | "패키지명 또는 경로를 지정해주세요" 안내 |
| 패키지를 못 찾음 | 검색 결과 후보를 보여주고 선택 요청 |
| 이미 같은 이름 스킬 존재 | 덮어쓰기 확인 |
| 패키지에 코드가 없음 | "분석할 코드가 없습니다" 안내 |
| 패키지가 매우 큼 (100+ 파일) | 핵심 public API만 요약, 상세는 "패키지 코드를 직접 읽어라" 지시 |

## 경계

**한다:**
- 패키지/폴더 탐색 + 코드 분석
- 사용자와 대화로 스킬 범위 확정
- `.claude/skills/`에 SKILL.md 생성

**안한다:**
- 글로벌 스킬 생성 (~/.claude/skills/)
- 패키지 코드 수정
- asmdef 수정
- 코드 구현 (스킬 생성만)
