# /ft:build — 피처 구현

Feature.md를 읽고, 의존 피처의 API를 파악한 뒤, 기존 코드를 활용하여 구현한다.

## 왜 이 단계가 중요한가

이 스킬의 존재 이유는 단 하나: **기존에 만든 기능을 무시하지 않고 활용하는 것.**
Feature.md의 의존성을 따라가며 기존 API를 읽고, 그걸 사용해서 코드를 작성한다.
이 과정 없이 구현하면 이미 있는 기능을 처음부터 다시 만들게 된다.

## 실행 흐름

### 플래그

| 플래그 | 설명 |
|--------|------|
| `--review` | 구현 완료 후 자동으로 `/ft:review --fix` 실행. 5단계(경험 학습) 전에 삽입됨 |

### 0단계: Pre-Execution Gate
구현 대상 피처 폴더에 Feature.md가 있는지, 그리고 `.featurecraft/tasks/{feature}-tasks.md`가 있는지 확인한다.

| Feature.md | tasks.md | 처리 |
|------------|----------|------|
| 있음 | 있음 | 정상 흐름 (태스크 기반 진행) |
| 있음 | 없음 | 구버전 피처로 간주 — 태스크 없이 전통 방식으로 진행 (진행률 섹션 생략) |
| 없음 | — | "Feature.md가 없습니다. `/ft:roadmap`으로 먼저 기획하세요." 안내 후 중단 |

### 1단계: Feature.md + 경험 학습 읽기
현재 피처의 Feature.md를 읽고 파악한다:
- 용도 — 뭘 만드는 건지
- 의존성 — 어떤 피처를 참고해야 하는지
- 구조 — 이미 만들어진 파일이 있는지
- API — 이미 정의된 공개 인터페이스가 있는지
- 주의사항 — 구현 시 조심할 점

`.featurecraft/learnings/` 폴더가 있으면 파일들을 읽어서 참고한다.
이전에 이 프로젝트에서 발견된 패턴이나 주의사항이 기록되어 있다.
예: "이 프로젝트에서 async는 UniTask 사용", "DB 호출은 항상 try-catch"

### 2단계: 의존성 체이닝
Feature.md의 의존성 섹션에 적힌 피처들을 순서대로 탐색한다:

1. 상대 경로로 의존 피처의 Feature.md 찾기
2. 해당 Feature.md의 API 섹션 읽기
3. API 옆에 적힌 파일 경로의 실제 코드 읽기 (시그니처, 반환 타입, 예외 등 확인)
4. 의존 피처의 의존성은 따라가지 않음 (1단계 깊이만)

**중복 읽기 방지:** 같은 대화에서 이미 읽은 파일은 다시 읽지 않는다. 컨텍스트에 내용이 남아있으면 그대로 사용한다. 파일이 변경되었을 가능성이 있을 때만 다시 읽는다.

사용자가 "바인딩 써서" 같이 명시한 경우:
1. 프로젝트에서 해당 피처 Feature.md 검색 (Glob `**/Feature.md` + 내용 매칭)
2. 찾으면 → 위와 동일하게 API 파악
3. 못 찾으면 → "해당 피처를 못 찾았습니다. 경로를 알려주시겠습니까?" 질문

### 3단계: 태스크 로드 + 진행률 요약 + 대상 태스크 결정

tasks.md가 **있는 경우에만** 이 단계를 실행한다 (없으면 4단계로 직행).

#### 3-A. tasks.md 파싱
`.featurecraft/tasks/{feature}-tasks.md`를 읽어 아래를 추출한다:
- Phase 목록 (`## Phase N:` 헤더)
- 태스크 목록 (각 체크박스 `- [ ]` 또는 `- [x]`)
- 각 태스크의 난이도/분량 메타 (`<난이도: X, 분량: Y>`)

#### 3-B. 진행률 요약 출력

```
📋 {피처명} 진행 상태
  Phase 1: 기반          3/5 완료 (60%) ━━━━━━───
  Phase 2: 통합          0/3 완료 (0%)  ──────────
  전체                   3/8 완료 (38%)

➡️ 다음 태스크: {첫 [ ] 태스크명}  <난이도: 보통, 분량: 2파일>
```

#### 3-C. 대상 태스크 결정

- **기본**: 파일의 첫 `[ ]` 태스크를 선택
- **사용자 지시 override**: 사용자가 "N번 먼저", "XX 태스크 먼저", "Phase 2부터" 같이 명시적으로 지시하면 해당 태스크 선택
- **전부 구현 요청**: 사용자가 "전부 구현해줘", "다 해줘" 등을 명시하면 남은 모든 태스크를 순차적으로 진행 (단, 태스크 간 보고는 각각)

남은 태스크가 없으면 → "✅ 모든 태스크가 완료되었습니다" 안내 후 중단.

### 4단계: 구현
파악한 정보를 기반으로 **대상 태스크 1개만** 구현한다. (여러 태스크를 한 번에 구현하지 않음, 사용자가 명시적으로 요청한 경우 제외)

- 의존 피처의 API를 **직접 사용**하여 구현 (새로 만들지 않음)
- Feature.md에 서브폴더 구조가 기술되어 있으면 **그 구조를 따라** 서브폴더를 생성하고 파일을 배치한다
- Feature.md에 서브폴더 언급이 없으면 피처 폴더 루트에 플랫하게 배치한다
- 네이밍, 구조는 기존 프로젝트 패턴을 따름
- tasks.md가 없는 구버전 피처는 Feature.md의 `포함 기능`을 기반으로 전체 구현

### 5단계: 태스크 체크박스 갱신

tasks.md가 있는 경우에만 실행.

- **완전히 완료된 태스크**만 `[ ]` → `[x]`로 갱신한다
- 부분 완료 / 중단된 경우 `[ ]` 유지하고 원인을 진행 보고에 기록
- 태스크에 붙은 `(초안)` 표시는 구현 과정에서 실제 분량/난이도가 명확해졌으면 제거 (선택)
- **구현 중 발견한 추가 태스크**가 있으면 tasks.md 해당 Phase 끝에 새 `[ ]` 항목으로 추가하고 진행 보고에 명시

### 6단계: Feature.md 업데이트 + 피처 인덱스 갱신
구현 완료 후 Feature.md를 최신 상태로 업데이트한다:

- **구조** 섹션: 생성된 파일 목록 + 한 줄 설명
- **API** 섹션: 외부에서 참조 가능한 public 메서드 + 파일 경로
- **의존성** 섹션: 새로 추가된 의존성 반영
- **상태**: 필요 시 wip → stable 등 변경
- `.featurecraft/FEATURE_INDEX.md` 갱신

### 7단계: Feature.md ↔ 코드 자동 검증
구현 완료 후, Feature.md의 **구조**와 **API** 섹션이 실제 코드와 일치하는지 자동 검증한다:

1. Feature.md의 **구조** 섹션에 나열된 파일 목록 vs 실제 폴더의 파일 목록 비교
   - 코드에 있는데 Feature.md에 없는 파일 → "추가됨" 표시
   - Feature.md에 있는데 코드에 없는 파일 → "삭제됨" 표시
2. Feature.md의 **API** 섹션에 적힌 메서드/클래스가 실제 코드에 존재하는지 Grep으로 확인
   - 시그니처가 변경됨 → "변경됨" 표시
   - API가 삭제됨 → "삭제됨" 표시
   - 새 public API가 추가됨 → "추가됨" 표시
3. 불일치가 있으면 Feature.md를 자동 갱신하고 변경 내역을 사용자에게 알린다

이 단계는 **항상 실행**된다 (플래그 불필요).

### 8단계: 자동 리뷰 (`--review` 플래그 시)
`--review` 플래그가 있으면 Feature.md 업데이트 직후, `/ft:review --fix`를 실행한다:
- 리포트 출력 (Critical / Warning / Info)
- Critical 항목은 자동 수정 (버그, 금지 패턴 위반)
- Warning 항목은 AskUserQuestion으로 각각 확인
- Info 항목은 리포트만, 수정 없음
- 리뷰 완료 후 9단계로 진행

### 9단계: 가이드 스킬 갱신 체크
수정한 파일이 **외부 패키지** (Packages/, unity-packages/ 등)에 속하면,
해당 패키지에 대한 가이드 스킬이 `.claude/skills/`에 있는지 확인한다.

1. Glob `.claude/skills/*/SKILL.md`로 기존 가이드 스킬 목록 파악
2. 각 스킬의 `패키지 루트` 경로와 수정된 파일 경로를 비교
3. 매칭되는 가이드 스킬이 있으면:
   - 스킬의 **피처 맵**에 없는 새 모듈/Feature.md가 추가됐는지 확인
   - **파이프라인**에 영향을 주는 변경인지 확인 (새 단계 추가, 기존 단계 파일 변경 등)
   - 갱신이 필요하면 해당 섹션만 업데이트하고 사용자에게 알림:
     "ℹ️ `{스킬명}` 가이드 스킬의 피처 맵/파이프라인을 갱신했습니다."
   - 갱신 불필요하면 아무것도 하지 않음 (무음)
4. 매칭되는 가이드 스킬이 없으면 → 스킵

### 10단계: 경험 학습 저장
구현 중 어려운 문제를 해결했거나, 프로젝트 특유의 패턴을 발견했으면
`.featurecraft/learnings/`에 기록한다. (폴더가 없으면 생성)

저장 기준 — 아래 3가지를 모두 만족할 때만:
- 구글링 5분으로 찾을 수 없는 것
- 이 프로젝트에 특화된 것
- 다음에도 적용할 수 있는 것

예시: "UniTask 사용 시 CancellationToken 항상 전달해야 에디터 종료 시 누수 방지"
반례: "Dictionary에 TryGetValue 쓰기" → 너무 일반적, 저장 안 함

### 11단계: 진행 보고 + 다음 태스크 프리뷰

#### 11-A. 진행 보고 (표준 템플릿)

tasks.md가 있는 경우:
```
✅ 완료: {태스크명}
   - 수정 파일: {파일 목록}
   - 핵심 변경: {1~2줄 요약}
   - Feature.md: {갱신 여부}

📋 진행률: {Phase명} — {완료}/{전체} ({%}), 전체 {완료}/{전체} ({%})
```

tasks.md가 없는 경우 (구버전 피처):
```
✅ 구현 완료
   - 수정 파일: {파일 목록}
   - 핵심 변경: {1~2줄 요약}
```

#### 11-B. Phase 경계 알림

방금 완료한 태스크가 **Phase의 마지막 태스크**였다면:
```
🎉 Phase {N} 완료! 다음은 Phase {N+1}: {이름} 입니다.
```

#### 11-C. 다음 태스크 프리뷰

남은 태스크가 있으면:
```
➡️ 다음 태스크: {다음 태스크명}
   난이도: {X}, 분량: {Y}
   예상: {간단 한 줄 설명}

`/ft:build`를 다시 실행하면 이 태스크를 진행합니다.
```

남은 태스크가 없으면:
```
🎊 모든 태스크 완료!
   - 리뷰: /ft:review
   - 커밋: /ft:push
```

tasks.md 없는 구버전 피처:
```
- 리뷰: /ft:review
- 커밋: /ft:push
```

## 태스크 단위 진행의 핵심 원칙

- **한 번에 한 태스크** — 기본은 대상 태스크 1개만 구현. 사용자가 "전부" 지시하면 반복
- **완료 기준 엄격** — 태스크 내의 모든 파일/변경이 완료되어야 `[x]`. 부분 완료는 `[ ]` 유지
- **순서 override 허용** — 사용자가 명시하면 순서 무시. 다만 의존 태스크를 건너뛰면 경고
- **태스크 추가 발견** — 구현 중 새 태스크 필요성을 발견하면 tasks.md에 추가 후 진행 보고에 명시
- **체크박스 신뢰** — 사용자가 외부에서 체크박스를 수정했으면 그 상태를 신뢰 (완료/미완료 간주)

## 의존성 체이닝 상세

### 탐색 깊이
- **1단계만** — 직접 의존하는 피처의 Feature.md + 실제 코드만 읽음
- 의존의 의존(2단계)은 읽지 않음 (토큰 절약 + 과도한 컨텍스트 방지)

### 실제 코드 읽기
Feature.md의 API 섹션에 파일 경로가 있으면:
```markdown
## API
- InventoryService.AddItem(itemId, count) → Scripts/InventoryService.cs
```
→ `Scripts/InventoryService.cs`를 직접 읽어서 정확한 시그니처/반환 타입/예외 처리 확인
→ Feature.md는 지도, 실제 코드가 진짜 스펙

### 순환 감지
의존성 탐색 중 이미 읽은 Feature.md를 다시 만나면:
- 경고 출력: "순환 의존성 감지: A → B → A"
- 해당 체인 중단 (무한 루프 방지)
- 사용자에게 알림

## 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| Feature.md 없음 | /ft:roadmap 안내 후 중단 |
| tasks.md 없음 | 구버전 피처로 간주 — 진행률 섹션 생략, 전통 방식 진행 |
| tasks.md 모든 태스크 완료 | "모든 태스크 완료" 안내 후 중단 |
| 태스크 부분 완료/중단 | `[ ]` 유지 + 진행 보고에 원인 기록 |
| 사용자가 체크박스 직접 수정 | 수정된 상태 신뢰 (완료 간주 시 다음 태스크로 진행) |
| 사용자가 "N번 먼저" 지시 | 순서 override — 해당 태스크로 점프 |
| 의존 태스크 건너뛰기 시도 | 경고 + 사용자 확인 후 진행 |
| 구현 중 추가 태스크 발견 | tasks.md 해당 Phase 끝에 새 `[ ]` 추가 + 진행 보고에 명시 |
| 의존 피처의 Feature.md 없음 | "/ft:scan으로 등록하시겠습니까?" 안내 |
| 의존 피처의 API 파일 경로가 틀림 | 경고 + Grep으로 해당 클래스/메서드 검색 시도 |
| deprecated 피처 의존 | 경고 + 대체 피처 제안 |
| 상대 경로 해석 실패 | 경고 + 경로 후보 제안 |
| 상위 Feature.md 존재 | 하위 피처 목록에 새 항목 추가 |

## 경계

**한다:**
- Feature.md 읽기 + 의존성 체이닝
- 의존 피처의 실제 코드 읽기 (API 파악)
- **tasks.md 로드 + 진행률 요약 + 대상 태스크 결정**
- 기존 API를 활용한 코드 구현 (**한 번에 한 태스크**)
- 피처 폴더 내 파일/폴더 생성
